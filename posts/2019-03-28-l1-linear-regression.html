L1L_1L1​ linear regression


I read an article on the errors in visualization. The example of forcing a relationship by cherry-picking scales is delightful. I recommend reading it.

I am interested in how to mislead people as much as possible, while being absolutely correct. The article inspires the following problem. Given 2 vectors x,y∈Rn\bm{x},\bm{y}\in \R^nx,y∈Rn. Let 1\bm{1}1 be the all 111 vector in Rn\R^nRn. We are interested in finding a,b∈Ra,b\in \Ra,b∈R, such that we minimizes ∥y−(ax+b1)∥p\|\bm{y}-(a\bm{x}+b\bm{1})\|_p∥y−(ax+b1)∥p​. Here ppp is either 1,21,21,2 or ∞\infty∞.

Note the problem is precisely the same as the linear regression problem. In the linear regression problem, we are given points (x1,y1),…,(xn,yn)(x_1,y_1),\ldots,(x_n,y_n)(x1​,y1​),…,(xn​,yn​) and we are interested in find a line f(x)=ax+bf(x) = ax+bf(x)=ax+b, such that it minimizes the error, defined as

∑i=1n∥yi−f(xi)∥p\displaystyle \sum_{i=1}^n \|y_i - f(x_i)\|_p i=1∑n​∥yi​−f(xi​)∥p​

For p=2p=2p=2, there is a O(n)O(n)O(n) time algorithm because there is a closed formula. For p=∞p=\inftyp=∞, the problem can be rewritten as a linear program with 333 variables and nnn constraints. Using Megiddo's result [1], there is a O(n)O(n)O(n) time algorithm to solve this problem.

Strangely, I could not find just how hard is it to solve the problem when p=1p=1p=1. This case is called the least absolute deviations. Statisticians just don't care about worst case running time as CS people do.

There are a few methods I found. One is to write it as a linear program on n+2n+2n+2 variables and nnn constraints and solve it using the simplex method. There are a bunch of other algorithms that specializes the simplex algorithm on this particular problem.

One can realize in the optimal solution, it is a line that goes through two of the points. The native algorithm can try all possible O(n2)O(n^2)O(n2) lines. For each line, the algorithm can compute the error in O(n)O(n)O(n) time. Together it is a O(n3)O(n^3)O(n3) time algorithm. There is a smarter algorithm. The optimal line that contains the point can actually be found in O(n)O(n)O(n) time. Indeed, it should be a line with the same number of points above as the same number of points below. Hence one can show the running time is O(n2)O(n^2)O(n2). This is basically the idea of [2], but their algorithm design is closer to the simplex algorithm. As far as I know, this seems to be the state of the art. There are many iterative methods, which of course depends on the input values.

Does a faster algorithm exist?
